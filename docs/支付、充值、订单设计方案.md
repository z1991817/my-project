# 支付、充值、订单设计方案

日期：2026-03-28

适用范围：当前 Node.js + Express + tsoa + MySQL + Redis 项目

## 一、方案结论

结合当前代码，这一版建议聚焦“积分充值闭环”，不要扩成完整电商系统。

当前已经具备的基础：

- `users.points` 已存在，可作为积分余额
- 注册成功会赠送积分
- `/app/me` 已返回用户积分
- 生图相关接口已经基于积分做余额校验、预扣、确认、释放

本次业务口径统一为：

- 注册赠送 `1000` 积分
- 充值 `9.9` 元得 `1000` 积分
- 充值 `39.9` 元得 `4500` 积分
- 充值 `99` 元得 `15000` 积分

## 二、这三个套餐需不需要维护

结论：需要维护，但第一版只需要在后端代码中静态维护，不需要先做数据库表和后台管理。

原因：

- 当前只有 3 个固定套餐，数量少、规则简单
- 目前项目还没有支付、订单、积分流水模块，先做套餐后台会扩大范围
- 套餐金额和积分必须以后端为准，本来就不能信任前端传值
- 第一版用静态配置最稳，开发成本最低

第一版推荐做法：

- 后端维护固定套餐常量
- 前端通过接口获取套餐列表
- 创建订单时前端只传 `packageId`
- 后端根据 `packageId` 生成订单金额、积分、套餐名称
- 支付回调时按订单快照发积分

只有在下面场景出现时，再升级为数据库维护：

- 套餐需要频繁调价
- 需要后台上下架
- 需要活动赠送、限时优惠、渠道差异价
- 套餐数量明显增加

## 三、和当前代码保持一致的调整

当前代码应统一为：

- `src/controllers/AuthController.ts` 注册赠送积分：`1000`
- `src/models/user.ts` 创建用户默认积分：`1000`
- `sql/users.sql` 用户表默认积分：`1000`

说明：

- 这样新注册用户、数据库默认值、接口返回口径一致
- 老用户是否补齐到 `1000`，建议按活动范围做一次补偿，不建议直接全表无差别更新

## 四、第一版建议的数据设计

### 1. `users`

保留现有字段：

- `id`
- `username`
- `email`
- `points`
- `status`
- `created_at`
- `updated_at`

说明：

- `points` 表示当前积分余额

### 2. `payment_orders`

用途：充值订单主表。

建议字段：

- `id`
- `order_no`
- `user_id`
- `order_type`，固定 `recharge`
- `package_id`
- `package_name`
- `amount`
- `points`
- `status`
- `payment_channel`
- `third_party_order_no`
- `paid_at`
- `created_at`
- `updated_at`

关键点：

- 即使套餐是静态配置，也要把金额、积分、套餐名写进订单
- 历史订单不能依赖实时套餐配置回算

### 3. `payment_transactions`

用途：支付流水表。

建议字段：

- `id`
- `order_id`
- `user_id`
- `channel`
- `transaction_type`
- `trade_no`
- `status`
- `request_data`
- `response_data`
- `callback_data`
- `created_at`
- `updated_at`

### 4. `points_logs`

用途：积分流水明细。

建议字段：

- `id`
- `user_id`
- `change_type`
- `change_amount`
- `balance_after`
- `order_id`
- `remark`
- `created_at`

`change_type` 建议值：

- `register_bonus`
- `recharge`
- `consume`
- `refund`

## 五、第一版套餐配置方式

推荐直接放在后端常量中，例如：

```ts
export const RECHARGE_PACKAGES = [
  { id: 'recharge_9_9', name: '1000积分', amount: 9.9, points: 1000 },
  { id: 'recharge_39_9', name: '4500积分', amount: 39.9, points: 4500 },
  { id: 'recharge_99', name: '15000积分', amount: 99, points: 15000 },
] as const;
```

接口约束：

- 前端只认 `packageId`
- 后端不接受前端直接传金额和积分作为最终值

## 六、业务流程

### 1. 用户充值流程

1. 用户请求充值套餐列表
2. 用户选择套餐并提交 `packageId`
3. 后端根据静态套餐配置创建订单
4. 后端调用支付渠道下单
5. 用户完成支付
6. 支付平台异步回调后端
7. 后端验签、校验订单号、校验金额、校验状态
8. 事务内更新订单为 `paid`
9. 给用户增加积分
10. 写入积分流水和支付流水

### 2. 和现有积分消费逻辑的关系

当前已有的积分扣减逻辑保持不变：

- 生图消费继续从 `users.points` 扣减
- 充值成功后只需要给 `users.points` 增加对应积分
- `points_logs` 负责补齐账本能力

## 七、幂等与一致性

支付回调必须满足：

1. 验签
2. 校验订单号
3. 校验金额
4. 校验订单当前状态
5. 做幂等控制
6. 在同一个事务中完成：
   订单改为 `paid`
   用户加积分
   写入积分流水
   写入支付流水

原则：

- 不允许重复回调重复加积分
- 不允许“订单已支付但积分没到账”
- 不允许“积分到账了但订单还是待支付”

## 八、Redis 建议

Redis 继续作为辅助组件，不做最终数据真相来源。

可用于：

- 创建订单防重复锁：`pay:create:lock:{userId}:{packageId}`
- 支付回调幂等锁：`pay:callback:lock:{orderNo}`
- 订单状态短缓存：`pay:order:status:{orderNo}`

MySQL 负责：

- 订单最终状态
- 用户积分余额
- 积分流水

## 九、接口建议

### 用户端

- `GET /app/recharge/packages`
  获取固定充值套餐列表
- `POST /app/recharge/orders`
  根据 `packageId` 创建充值订单
- `GET /app/recharge/orders`
  获取我的充值订单列表
- `GET /app/recharge/orders/{id}`
  获取我的充值订单详情
- `GET /app/points/logs`
  获取我的积分流水
- `GET /app/me`
  获取当前用户信息，包含积分余额

### 支付回调

- `POST /api/payment/callback/{channel}`
  支付渠道异步回调

### 管理端

- `GET /api/v1/admin/orders`
- `GET /api/v1/admin/orders/{id}`
- `GET /api/v1/admin/customers`
- `GET /api/v1/admin/customers/{id}`
- `GET /api/v1/admin/customers/{id}/points-logs`

第一版不做：

- `recharge_products` 后台管理接口

## 十二、最终建议

这版方案建议按下面原则执行：

- 注册赠送积分统一为 `1000`
- 充值套餐固定 3 档：`9.9/1000`、`39.9/4500`、`99/15000`
- 第一版套餐只在后端静态维护，不单独做后台维护

### 3. 已完成的自测结果

已验证通过的主流程：

- 用户登录
- 获取充值套餐列表
- 创建充值订单
- 用户发起 `mock-pay`
- 通过 `mock` 回调完成支付
- 支付成功后积分到账
- 查询我的订单列表
- 查询我的订单详情
- 查询我的积分流水
- 管理端查询订单
- 管理端查询客户
- 管理端查询客户积分流水
- 新注册用户赠送 `1000` 积分
- 新注册用户 `points_logs` 中存在 `register_bonus`

关键验证结果：

- 非法套餐创建订单返回 `404`
- 非本人查看订单详情返回 `404`
- 同一订单重复支付不会重复加积分
- 已支付订单再次回调不会重复入账
- `payment_transactions` 与 `points_logs` 会按订单正确落一条记录

### 4. 已完成的并发压测结果

本地已做 1 轮针对性并发压测：

- 同一订单并发支付：10 个并发请求
- 同一套餐并发创建订单：5 个并发请求

结果：

- 同一订单并发支付后，积分只增加 `1000`
- 同一订单只写入 `1` 条 `payment_transactions`
- 同一订单只写入 `1` 条 `points_logs`
- 10 个并发支付请求中，成功表现为：
  - 1 次“模拟支付成功”
  - 多次“订单已支付”
  - 多次“订单正在处理中，请稍后重试”
- 5 个并发创建订单请求中：
  - 1 次成功
  - 4 次返回 `429`

结论：

- 当前幂等和防重复锁在本地压测下有效
- 当前支付闭环没有出现重复入账
