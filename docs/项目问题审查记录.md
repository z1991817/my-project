# 项目问题审查记录

日期：2026-03-25

范围：本次仅做只读审查，不修改代码。重点检查了入口层、鉴权、控制器、模型层以及图片生成相关流程。

## 结论概览

当前项目可以通过 `npm run build`，说明 TypeScript 编译和 tsoa 路由生成在当前状态下没有直接报错。

主要问题集中在：

- 鉴权边界不清，存在提权风险
- 写接口缺少权限控制
- 会话数据缺少用户归属校验
- 个别列表接口实现存在明显逻辑错误
- 限流实现有并发穿透风险
- 默认安全配置偏宽松

## 详细问题

### 1. 普通用户 Token 可穿透管理员鉴权

严重级别：高

问题说明：

- 普通用户登录和管理员登录使用同一个 `JWT_SECRET`
- `adminJwt` 鉴权只验证 token 签名是否正确
- 未校验 token 是否是管理员签发，也未校验 `role` 或 token 类型

影响：

- 普通用户 token 可能被当作管理员 token 使用
- 会导致后台接口越权访问，属于提权问题

### 2. 图片写接口未做鉴权

严重级别：高

问题说明：

- `/app/images` 的新增、更新、删除接口都没有 `@Security`
- 当前实现等于任何能访问接口的人都可以改图库数据

影响：

- 可被未登录用户直接写入、篡改、删除图片记录
- 属于直接暴露的后台写权限问题

### 3. 会话历史缺少用户归属校验

严重级别：高

问题说明：

- `session_id` 只校验 UUID 格式
- 读取和更新对话历史时，按 `session_id` 直接查询
- 会话记录没有体现用户归属约束
- 接口在继续会话时也没有校验当前登录用户是否拥有该会话

影响：

- 已登录用户拿到其他人的 `session_id` 后，可能读取上下文并续写对话
- 还可能覆盖目标会话最后一条 assistant 消息

### 5. 邮箱验证码限流逻辑存在并发穿透

严重级别：中

问题说明：

- 当前流程是先 `redis.exists()` 判断，再执行发送，再调用 `checkRateLimit()`
- `checkRateLimit()` 本身是原子接口，但返回值没有参与前置控制
- 并发请求时，多个请求可能同时通过 `exists()` 检查

影响：

- 同一邮箱可能被短时间重复发送验证码
- 会增加邮件成本，也容易被滥用

代码位置：

### 6. CORS 默认全开放

严重级别：中低

问题说明：

- 应用入口直接使用 `app.use(cors())`
- 没有配置允许来源、方法、Header 范围

影响：

- 当前如果只做简单 Bearer Token，风险还算可控
- 但一旦后续加入 cookie、浏览器凭证、管理后台域名隔离等需求，安全边界会比较弱
